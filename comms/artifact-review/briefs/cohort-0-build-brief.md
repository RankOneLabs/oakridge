## Decisions made

- **Status enum unification.** Both `plans.status` and `handoff_docs.status` use `pending_approval | approved | rejected | superseded`. The old proposal store's `pending|approved|rejected|failed` is dropped (failed collapses into rejected + reason; pending into pending_approval). **Rationale:** one vocabulary across both artifact types makes the freezing, webhook payloads, and PWA logic uniform; rejected-with-reason already carries failure semantics.

- **Move webhook emit from route handler into `submitPhaseHandoff()`.** The current `enqueueWebhook` call at `safir/src/api/routes/phases.ts:51` moves inside `submitPhaseHandoff()` and runs inside the function's transaction with the handoff insert and atom_edits seed. **Rationale:** keeps the atom_edits seed, the canonical insert, and the webhook envelope as one atomic unit; partial states (e.g., a webhook fired but no atom_edits seed because of a crash) are impossible.

- **`handoff_docs` table keeps its name; routes use `/build-briefs/`.** No invasive rename. **Rationale:** the spec calls this out as a deliberate trade â€” internal rename is churn without value once the surface is consistent.

- **`prev_value` mandatory on atom_edits writes; 409 returns full canonical state.** Body shape on conflict: `{error: 'stale_prev_value', current_value, latest_edit_id, edited_by, created_at}`. No auto-retry. **Rationale:** CAS handles operator/agent races; agents surface conflicts in their reply rather than racing the operator (locks in cohort 1's contract).

- **Approve does not block on open threads.** The status PATCH succeeds with open threads present; cohort 2's PWA warns. Threads on approved artifacts remain readable but accept no new messages or pings (409). **Rationale:** explicit operator choice; threads-as-history is more useful than threads-as-blockers.

- **Reopen does not auto-cancel in-flight builds.** Operator decides. **Rationale:** build runs are expensive; auto-cancel could destroy work; warning is the safer default.

- **Plan-tasks (commits) are not first-class safir rows.** They live in `plan_cohorts.notes` (markdown) and are structured by planner-2 into the build brief. **Rationale:** scoped to v1; per-commit rows are reversible later when a real reason appears.

- **Initial atom_edits batch is part of the artifact-creation transaction.** NĂ—3 + M rows for plans (cohort attrs + edge presence markers); one row per atom for build briefs. `prev_value=NULL`, `edited_by='agent-initial'`. **Rationale:** keeps the atom_edits log a complete history from emission onward.

- **Migrations 006 + 007 land in the same PR but as separate files in the order shown.** `atom_edits.target_type` references both `'plan'` and `'build_brief'`; splitting would force a midpoint where atom_edits exists but plans don't, or vice versa. **Rationale:** spec explicitly rejects the split.

- **Edge atoms are encoded as `edge:<from>-><to>` with value `'1'`.** Stored as a presence marker so deletes are conflict-aware via the same CAS path as cohort/title edits. **Rationale:** uniform anchor handling; the value content is irrelevant since the row's existence is what's meaningful, and CAS protects against concurrent delete-then-re-add.

- **`materializePlanCohortsAsTasks` writes to existing `tasks` and `dependencies` tables.** Each cohort produces one `tasks` row with `parent_id = plan.parent_task_id`, `title`, `notes`, `priority`, `status='backlog'`, `project_id` inherited from the parent. Each cohort_dependency produces one row in `dependencies` (existing table from migration 001). **Rationale:** reuses existing dependency machinery; no new join surface; one transaction either fully materializes or nothing does.

- **Migration script lives in `oakridge/kbbl/scripts/`.** TypeScript, runs against the kbbl data dir on the operator's machine. **Rationale:** the script reads kbbl's filesystem store, so it belongs with kbbl; cleaner than a Python script that has to know about kbbl's data layout.

- **Punt #7 resolved: replace planning-proposals fetches in-place in `App.tsx`.** There are exactly 4 call sites in `App.tsx` (lines 631, 4215, 4243, 4251); proxied via small `safir-proxy.ts` extension. **Rationale:** 4 sites is well under the "scattered" threshold; the proxy extension is trivial; avoids leaving a dead path-rewrite rule.

- **`safir-plan` binary renamed to `safir-decompose`.** Updates `pyproject.toml` script entry and `argparse` `prog=`. **Rationale:** the spec's exit criterion uses `safir-decompose <task-id>` as a literal CLI invocation â€” that's a spec requirement, not just vocabulary.

- **Tool renames in planner-1: `create_task` â†’ `create_cohort`, `add_dependency` â†’ `add_cohort_dependency`.** **Rationale:** matches the new artifact vocabulary; the agent's trace stays legible against the new domain.

- **Cohort dependencies use `cohort_index` (integer) not `cohort_id`.** Cohort indices are stable within a plan (PK is `(plan_id, cohort_index)`). Cohorts inserted via add/split during review (cohort 1) extend the index space â€” `cohorts[N+1]`, gaps allowed. **Rationale:** the spec's V1 position; revision history reads more naturally with stable indices.

- **`ProposalReviewView` stays alive in App.tsx during this PR.** Its four fetches get rewired to safir; the component is deleted in cohort 2 when `PlanReviewView` replaces it. **Rationale:** deleting it here would leave the PWA without a plan review surface for the cohort-1/cohort-2 gap; the bridge cost is one PR's worth of churn.

- **Plan task notes are markdown, not structured JSON.** The spec says "the cohort's `notes` field carries both the cohort scope description and the breakdown of plan-tasks (commits)." Stored as a single `TEXT NOT NULL` column. **Rationale:** matches existing `tasks.notes` convention; planner-2 parses by convention, not by schema.

- **`safir-py` does not need updating.** Despite the plan calling out "safir-py and kbbl consumers", grep confirms `safir-py` has zero references to `phase.handoff_submitted` or `WebhookEvent`. Only `kbbl/core/server/handlers/safir-webhook.ts` is updated. **Rationale:** verified against working tree; no consumer means no change needed.

- **Threads route listing path: `GET /artifacts/:target_type/:target_id/threads`** rather than `GET /atoms/:target_type/:target_id/threads`. **Rationale:** the `/atoms/` namespace is already overloaded between "atom map" and "atom history"; a third meaning would invite path-routing bugs in Hono. Spec's `/atoms/.../threads` phrasing is taken as descriptive, not literal.

## Approaches rejected

- **Collapse canonical artifact rows into atom_edits entirely.** Rejected because canonical rows are FK-referenced (`plan_cohorts` is read on approval; `handoff_docs.id` is the join target for runs); the as-emitted state is independently useful for revision-floor reads.

- **Block approve until open threads resolved.** Rejected: operator's explicit choice. Warning is enough; resolution is optional.

- **Per-commit safir rows.** Rejected for v1: rows would multiply by ~3-5Ă— and add a join surface for no immediate consumer.

- **Rename `handoff_docs` table to `build_briefs`.** Rejected: invasive, no value once the API surface is consistent.

- **Per-event webhook tables.** Rejected: existing `webhook_deliveries` handles all events fine; one queue is simpler.

- **Auto-resolve threads on artifact approval.** Rejected: threads stay open as historical record; freeze prevents new activity.

- **Path-rewrite `/planning-proposals` â†’ `/plans` in `safir-proxy.ts`.** Rejected per punt #7 resolution: 4 PWA sites is few enough that in-place rewrite is cleaner than a permanent rewrite rule.

- **Single combined endpoint `POST /threads/:id/agent-response/:status` (path-param the outcome).** Rejected in favor of body-param `{status: 'completed'|'failed'}` on a single `POST /threads/:id/agent-response` endpoint. **Rationale:** path-versioning the status forces two near-identical routes; one endpoint with a discriminator body is the existing safir pattern (mirrors `PATCH /:type/:id/status`).
